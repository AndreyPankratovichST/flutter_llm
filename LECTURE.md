## 1. Введение

**Flutter LLM** — это мобильное Flutter-приложение, позволяющее пользователю:

- **Искать** языковые модели (LLM) на платформе HuggingFace;
- **Скачивать** GGUF-файлы моделей на устройство с поддержкой возобновления загрузки;
- **Управлять** локально сохранёнными моделями (просмотр, удаление);
- **Загружать** модель в память и **вести чат** с ней в реальном времени (стриминг токенов).

Приложение построено по принципам **Clean Architecture**, использует **BLoC** для управления состоянием и **get_it** для внедрения зависимостей.

## 2. Основные понятия

Прежде чем разбирать код проекта, необходимо понять ключевые технологии и концепции, на которых он построен.

### 2.1. Большие языковые модели (LLM)

**LLM (Large Language Model)** — это нейронная сеть с большим числом параметров (от единиц до сотен миллиардов), обученная на огромных объёмах текстовых данных. Задача LLM — предсказывать следующий токен (фрагмент текста) на основе предыдущих.

Примеры известных LLM:
- **GPT** (OpenAI) — модели серии GPT-3, GPT-4;
- **LLaMA** (Meta) — семейство открытых моделей LLaMA 2, LLaMA 3;
- **Mistral**, **Gemma**, **Qwen** — другие открытые модели различных компаний.

Ключевое свойство LLM для данного проекта: многие модели имеют **открытые веса** — файлы с обученными параметрами, которые можно скачать и запустить локально, без обращения к облачным API. Именно это и делает Flutter LLM — скачивает файлы весов и выполняет генерацию текста прямо на устройстве пользователя.

### 2.2. Архитектура Transformer

Все современные LLM основаны на архитектуре **Transformer**, предложенной в статье "[Attention Is All You Need](https://arxiv.org/pdf/1706.03762)" (2017). Ключевой механизм — **self-attention** (механизм самовнимания), позволяющий модели учитывать связи между всеми словами во входной последовательности одновременно.

Основные компоненты Transformer:
- **Embedding** — преобразование токенов в числовые векторы;
- **Multi-Head Attention** — несколько параллельных «голов» внимания, каждая из которых учится отслеживать разные типы связей между словами;
- **Feed-Forward Network** — полносвязные слои после каждого блока внимания;
- **Layer Normalization** — нормализация для стабильного обучения.

Модель состоит из десятков таких слоёв (блоков), расположенных последовательно. Количество параметров определяется числом слоёв, размером скрытого состояния и количеством «голов» внимания. Например, модель с 7 миллиардами параметров (7B) содержит порядка 32 слоёв с размером скрытого состояния 4096.

В контексте данного проекта поле `architecture` в `ModelInfo` отражает конкретную реализацию Transformer (llama, mistral, phi и т.д.), а поле `parameters` — количество параметров модели.

### 2.3. Токены и токенизация

LLM работают не с целыми словами, а с **токенами** — фрагментами текста. Токенизатор разбивает входной текст на последовательность токенов перед подачей в модель.

Типичные примеры:
- Слово `"привет"` может быть одним токеном;
- Слово `"токенизация"` может разбиться на несколько: `"токен"`, `"из"`, `"ация"`;
- Знаки пунктуации и пробелы — тоже токены.

В среднем один токен соответствует примерно 3–4 символам на английском языке и 1–2 символам на русском. Каждый токен имеет числовой идентификатор в **словаре** (vocabulary) модели. Словарь обычно содержит 32 000–128 000 токенов.

Токенизация важна для понимания работы приложения: когда пользователь отправляет сообщение, оно сначала разбивается на токены, а ответ модели также генерируется **потокенно** — именно поэтому текст в чате появляется постепенно (стриминг).

### 2.4. Контекстное окно (Context Window)

**Контекстное окно** — максимальное количество токенов, которое модель может обработать за один раз. В него входит как входной запрос (промпт), так и сгенерированный ответ.

Примеры типичных размеров контекстного окна:
- **2 048 токенов** — ранние модели;
- **4 096–8 192 токенов** — большинство 7B-моделей;
- **32 768–131 072 токенов** — современные модели с расширенным контекстом.

Если разговор превышает размер контекстного окна, ранние сообщения «выпадают» из памяти модели. В данном проекте размер контекста отображается в `ModelInfoSheet` через поле `contextSize`, значение которого считывается из метаданных загруженной модели.

### 2.5. Инференс и стриминг генерации

**Инференс (inference)** — процесс генерации текста моделью. В отличие от обучения, инференс не изменяет веса модели, а использует их для предсказания.

Генерация работает **авторегрессионно**: на каждом шаге модель предсказывает один следующий токен, затем добавляет его к входу и повторяет. Этот процесс продолжается до тех пор, пока модель не выдаст специальный токен конца последовательности (EOS) или не будет достигнут лимит длины.

**Стриминг (streaming)** — подход, при котором каждый сгенерированный токен немедленно передаётся клиенту, не дожидаясь завершения всего ответа. В данном проекте стриминг реализован через Dart-овые `Stream`:
- `LLMService.sendQuery()` возвращает `Stream<String>`;
- `ChatBloc` подписывается на этот стрим и обновляет последнее сообщение при каждом новом фрагменте;
- В UI текст ответа «печатается» в реальном времени.

### 2.6. llama.cpp

**llama.cpp** — открытая C/C++ библиотека для запуска LLM на обычном пользовательском оборудовании. Разработана Георгием Герганым (Georgi Gerganov) и сообществом.

Ключевые возможности:
- **Поддержка CPU-инференса** — модель может работать без GPU, используя оптимизированные инструкции процессора (AVX, NEON для ARM);
- **Аппаратное ускорение** — поддержка GPU через Vulkan (кроссплатформенный), CUDA (NVIDIA), Metal (Apple);
- **Низкое потребление памяти** — благодаря квантизации и оптимизированным форматам хранения;
- **Кроссплатформенность** — работает на Windows, Linux, macOS, Android, iOS;
- **Поддержка множества архитектур** — LLaMA, Mistral, Phi, Gemma, Qwen и десятки других.

В данном проекте llama.cpp используется через Flutter-обёртку `llamadart`, которая предоставляет Dart API (`LlamaEngine`, `ChatSession`) для нативного C++-движка. В `pubspec.yaml` в секции `hooks` настраиваются нативные бэкенды:
```yaml
hooks:
  user_defines:
    llamadart:
      llamadart_native_backends:
        platforms:
          android-arm64:
            backends: [vulkan]
          linux-x64: [vulkan, cuda]
          windows-x64: [vulkan, cuda]
```

### 2.7. Формат GGUF

**GGUF (GPT-Generated Unified Format)** — бинарный формат файлов для хранения весов языковых моделей, разработанный в рамках проекта llama.cpp. Пришёл на смену устаревшим форматам GGML и GGJT.

Структура GGUF-файла:
- **Заголовок** — магическое число, версия формата;
- **Метаданные** — пары «ключ–значение» с информацией о модели: название, архитектура, количество параметров, тип квантизации, размер контекста и т.д.;
- **Тензоры** — собственно веса модели в заданном формате (тип данных, размерности, бинарные данные).

Преимущества GGUF:
- **Самодостаточность** — один файл содержит и веса, и все метаданные;
- **Кроссплатформенность** — формат не зависит от фреймворка (PyTorch, TensorFlow);
- **Поддержка квантизации** — веса хранятся в сжатом формате;
- **Расширяемость** — метаданные могут содержать произвольные ключи.

В данном проекте:
- Приложение ищет и скачивает именно `.gguf`-файлы с HuggingFace;
- `ModelStorageService` фильтрует локальные файлы по расширению `.gguf`;
- `LLMServiceImpl` передаёт путь к GGUF-файлу в `LlamaEngine.loadModel()`;
- Информация о модели (`ModelInfo`) извлекается из метаданных GGUF через ключи вида `general.name`, `general.architecture`, `general.parameter_count`, `general.file_type`.

### 2.8. Квантизация

**Квантизация** — метод сжатия модели путём уменьшения точности хранения весов. Вместо 32- или 16-битных чисел с плавающей запятой веса хранятся в формате с меньшей битностью.

Основные уровни квантизации:

| Обозначение | Бит на вес | Размер 7B-модели | Качество |
|---|---|---|---|
| F16 | 16 бит | ~14 ГБ | Максимальное |
| Q8_0 | 8 бит | ~7 ГБ | Почти без потерь |
| Q6_K | 6 бит | ~5.5 ГБ | Очень хорошее |
| Q5_K_M | 5 бит | ~5 ГБ | Хорошее |
| Q4_K_M | 4 бита | ~4 ГБ | Приемлемое |
| Q3_K_M | 3 бита | ~3.5 ГБ | Заметная деградация |
| Q2_K | 2 бита | ~2.5 ГБ | Значительная деградация |

Суффиксы в названиях квантизаций:
- **K** — метод k-quant, более умная группировка весов;
- **M** — medium (средний баланс размер/качество);
- **S** — small (компактнее, но ниже качество);
- **L** — large (больше, но выше качество).

Квантизация критически важна для мобильных устройств: без неё модель с 7 миллиардами параметров занимала бы ~14 ГБ в формате FP16, что недопустимо для большинства смартфонов. С квантизацией Q4_K_M та же модель занимает ~4 ГБ и при этом сохраняет приемлемое качество генерации.

В проекте на экране поиска модели пользователь видит список доступных GGUF-файлов — каждый файл соответствует конкретному варианту квантизации. Пользователь выбирает нужный баланс между размером файла и качеством модели. Тип квантизации также отображается в `ModelInfoSheet` через поле `quantization`.

### 2.9. HuggingFace

**HuggingFace** — крупнейшая платформа для хранения и распространения моделей машинного обучения. Выполняет роль «GitHub для ML-моделей».

Основные возможности:
- **Model Hub** — каталог с сотнями тысяч моделей, организованных по задачам (text-generation, text-to-image и т.д.);
- **REST API** — программный доступ к каталогу и файлам моделей;
- **Git LFS** — модели хранятся в Git-репозиториях с поддержкой больших файлов.

API HuggingFace, используемый в проекте:
- **Поиск моделей** — `GET https://huggingface.co/api/models?search=...&filter=gguf&sort=downloads` — возвращает JSON-массив моделей с метаданными (id, автор, количество скачиваний, лайков, тип задачи);
- **Получение информации о модели** — `GET https://huggingface.co/api/models/{id}` — возвращает детальную информацию, включая список файлов (`siblings`);
- **Скачивание файла** — `GET https://huggingface.co/{id}/resolve/main/{filename}` — прямая ссылка на скачивание конкретного файла.

В проекте класс `HuggingFaceApiServiceImpl` инкапсулирует работу с этим API, а `PipelineTag` определяет доступные фильтры по типу задачи.

## 3. Обзор используемых пакетов

### 3.1. `llamadart` (v0.6.4)

**Назначение:** Основной пакет проекта — Flutter-обёртка над библиотекой llama.cpp для запуска LLM-моделей в формате GGUF непосредственно на устройстве.

**Ключевые возможности:**
- `LlamaBackend()` — инициализация нативного бэкенда (CPU, Vulkan, CUDA);
- `LlamaEngine` — движок для загрузки модели, получения метаданных (`getMetadata()`), размера контекста (`getContextSize()`), информации о бэкенде (`getBackendName()`), а также поддержки vision/audio;
- `ChatSession` — сессия чата, через метод `create()` принимает список сообщений (`LlamaTextContent`) и возвращает `Stream` с результатом генерации по частям (chunk-by-chunk);
- Поддержка аппаратного ускорения через Vulkan и CUDA (настраивается в `pubspec.yaml` в секции `hooks`).

**Как используется в проекте:** Класс `LLMServiceImpl` создаёт `LlamaEngine`, загружает GGUF-модель, открывает `ChatSession` и возвращает стрим текстовых фрагментов ответа.

### 3.2. `flutter_bloc` (v9.1.1)

**Назначение:** Библиотека для управления состоянием приложения по паттерну BLoC (Business Logic Component).

**Ключевые возможности:**
- `Bloc<Event, State>` — базовый класс, принимающий события и эмитящий новые состояния;
- `BlocProvider` — InheritedWidget, предоставляющий экземпляр BLoC дочерним виджетам;
- `BlocBuilder` — виджет, перестраивающийся при изменении состояния;
- `BlocListener` — виджет для выполнения побочных эффектов (навигация, SnackBar) при смене состояния;
- `BlocSelector` — оптимизированный BlocBuilder, реагирующий только на изменение выбранного поля состояния.

**Как используется в проекте:** В приложении 5 BLoC-классов: `ChatBloc`, `ModelLoaderBloc`, `ModelListBloc`, `ModelSearchBloc` и `DownloadManagerBloc`. Каждый отвечает за свою область ответственности.

### 3.3. `get_it` (v9.2.1)

**Назначение:** Сервис-локатор (Service Locator) для внедрения зависимостей.

**Ключевые возможности:**
- `registerSingleton<T>()` — регистрация единственного экземпляра (Singleton);
- `registerFactory<T>()` — регистрация фабрики, создающей новый экземпляр при каждом запросе;
- `getIt<T>()` — получение зарегистрированного экземпляра по типу.

**Как используется в проекте:** В файле `app_di_config.dart` регистрируются все сервисы, репозитории, use cases и BLoC-классы. Сервисы и репозитории — как Singleton, BLoC-классы — как Factory (кроме `DownloadManagerBloc`, который живёт всё время жизни приложения).

### 3.4. `http` (v1.3.0)

**Назначение:** HTTP-клиент для выполнения сетевых запросов.

**Ключевые возможности:**
- `http.get()` — простые GET-запросы;
- `http.Client` с `client.send(Request)` — потоковая загрузка с доступом к `response.stream`;
- Поддержка HTTP-заголовков, в том числе `Range` для частичной загрузки (докачка файлов).

**Как используется в проекте:** `HuggingFaceApiServiceImpl` использует `http.get()` для запросов к API HuggingFace. `ModelStorageServiceImpl` использует `http.Client` с потоковым чтением `response.stream` для побайтовой загрузки файлов моделей с поддержкой заголовка `Range` для возобновления прерванных загрузок.

### 3.5. `path_provider` (v2.1.5)

**Назначение:** Кроссплатформенное определение директорий файловой системы устройства.

**Ключевые возможности:**
- `getApplicationDocumentsDirectory()` — путь к директории документов приложения (для iOS/macOS/Android/Windows/Linux).

**Как используется в проекте:** Определяет папку `models/` внутри директории документов приложения для хранения скачанных GGUF-файлов и файла метаданных загрузок.

## 4. Архитектура проекта: Clean Architecture

Проект структурирован по принципам **Clean Architecture**, что обеспечивает разделение ответственности и независимость слоёв.

```
lib/
├── core/                    # Утилиты уровня приложения
├── data/                    # Слой данных (реализации)
│   ├── repositories/        # Реализации репозиториев
│   └── services/            # Сервисы (абстракции + реализации)
│       └── impl/            # Конкретные реализации сервисов
├── di/                      # Внедрение зависимостей
├── domain/                  # Слой бизнес-логики (чистый Dart)
│   ├── core/                # Базовые типы (Result, UseCase)
│   ├── entities/            # Сущности предметной области
│   ├── repositories/        # Абстракции репозиториев
│   └── usecases/            # Варианты использования
├── presentation/            # Слой представления (UI)
│   ├── blocs/               # BLoC-компоненты
│   ├── extensions/          # Dart-расширения
│   ├── pages/               # Экраны приложения
│   ├── router/              # Навигация
│   ├── utils/               # Константы и размеры
│   └── widgets/             # Переиспользуемые виджеты
└── main.dart                # Точка входа
```

**Правило зависимостей:** Domain-слой не зависит ни от чего. Data-слой реализует абстракции из Domain. Presentation-слой зависит от Domain (через use cases) и не знает о Data-слое напрямую.

## 5. Слой Domain — бизнес-логика

### 5.1. Сущности (Entities)

Сущности — неизменяемые (immutable) классы данных, описывающие объекты предметной области. Они не зависят ни от каких фреймворков.

#### `Message`
```dart
class Message {
  final String text;
  final bool isUser;
}
```
Представляет одно сообщение в чате. Поле `isUser` определяет, является ли автор сообщения пользователем (`true`) или языковой моделью (`false`). Метод `copyWith` позволяет создать изменённую копию — это необходимо для обновления текста ответа модели по мере стриминга.

#### `LocalModel`
```dart
final class LocalModel {
  final String name;
  final String path;
  final int sizeBytes;
}
```
Описывает модель, сохранённую на устройстве. Содержит имя файла, полный путь и размер в байтах.

#### `HuggingFaceModel`
```dart
final class HuggingFaceModel {
  final String id;        // например "TheBloke/Llama-2-7B-GGUF"
  final String author;
  final int downloads;
  final int likes;
  final String? pipelineTag;
}
```
Модель из каталога HuggingFace. Поле `pipelineTag` указывает тип задачи (text-generation, text-to-image и т.д.).

#### `ModelFile`
```dart
final class ModelFile {
  final String filename;
  final int sizeBytes;
}
```
Один GGUF-файл внутри репозитория модели на HuggingFace. Используется при отображении списка доступных для скачивания квантизаций.

#### `ModelInfo`
```dart
final class ModelInfo {
  final String name;
  final String architecture;
  final int contextSize;
  final String backendName;
  final bool supportsVision;
  final bool supportsAudio;
  final String parameters;
  final String quantization;
}
```
Техническая информация о загруженной модели, извлекаемая из метаданных GGUF-файла через `LlamaEngine`. Отображается в bottom sheet на экране чата.

#### `DownloadTask` и `DownloadTaskMetadata`
```dart
final class DownloadTask {
  final String modelId;
  final String filename;
  final DownloadStatus status;  // pending | downloading | completed | failed
  final double progress;        // от 0.0 до 1.0
}
```
`DownloadTask` — текущая задача загрузки с прогрессом, используется BLoC-ом менеджера загрузок для отображения состояния в UI.

`DownloadTaskMetadata` — сериализуемые метаданные загрузки (modelId, filename, url, targetPath), сохраняемые в JSON-файл. Нужны для **возобновления** прерванных загрузок после перезапуска приложения.

#### `PipelineTag`
```dart
enum PipelineTag {
  textGeneration('text-generation', 'Text Generation'),
  textToImage('text-to-image', 'Text to Image'),
  textToVideo('text-to-video', 'Text to Video'),
  textToAudio('text-to-audio', 'Text to Audio');
}
```
Перечисление типов задач моделей. Используется как фильтр при поиске моделей на HuggingFace. На данный момент полностью поддерживается только `textGeneration` (загрузка GGUF-файлов), остальные типы отмечены как «Coming soon».

### 5.2. Репозитории (абстракции)

Абстрактные классы, определяющие контракт работы с данными. Реализации находятся в слое Data.

#### `LLMRepository`
```dart
abstract class LLMRepository {
  Future<void> initModel(String modelPath);
  Future<void> disposeModel();
  Stream<String> sendMessage(String text);
  Future<ModelInfo> getModelInfo();
}
```
Контракт для работы с языковой моделью: загрузка, выгрузка, отправка сообщения (со стримингом ответа), получение информации о модели.

#### `ModelRepository`
```dart
abstract class ModelRepository {
  Future<List<HuggingFaceModel>> searchModels(String query, {PipelineTag? pipelineTag});
  Future<List<ModelFile>> getModelFiles(String modelId);
  Stream<double> downloadModel(String modelId, String filename, {int startByte});
  String getDownloadUrl(String modelId, String filename);
  Future<String> getDownloadPath(String filename);
  Future<int> getPartialFileSize(String targetPath);
  Future<List<DownloadTaskMetadata>> getPendingDownloads();
  Future<void> saveDownloadMetadata(DownloadTaskMetadata metadata);
  Future<void> removeDownloadMetadata(String filename);
  Future<List<LocalModel>> getLocalModels();
  Future<void> deleteModel(String path);
}
```
Контракт для управления моделями: поиск на HuggingFace, скачивание с прогрессом, управление метаданными загрузок, работа с локальным хранилищем.

### 5.3. Варианты использования (Use Cases)

Каждый use case представляет **одну бизнес-операцию**. Это соответствует принципу единственной ответственности (SRP) из SOLID.

#### Use Cases для работы с LLM

| Use Case | Базовый класс | Параметр | Результат | Описание |
|---|---|---|---|---|
| `LLMInitUseCase` | `UseCase<Unit, String>` | путь к модели | `Unit` | Загружает GGUF-модель в память через `LLMRepository.initModel()` |
| `LLMDisposeUseCase` | `UseCaseNoParams<Unit>` | — | `Unit` | Выгружает модель из памяти, освобождает ресурсы |
| `LLMSendMessageUseCase` | `StreamUseCase<String, String>` | текст сообщения | поток строк | Отправляет сообщение модели и возвращает стрим фрагментов ответа |
| `GetModelInfoUseCase` | `UseCaseNoParams<ModelInfo>` | — | `ModelInfo` | Извлекает метаданные загруженной модели (архитектура, контекст и т.д.) |

#### Use Cases для управления моделями

| Use Case | Базовый класс | Параметр | Результат | Описание |
|---|---|---|---|---|
| `SearchModelsUseCase` | `UseCase<..., SearchModelsParams>` | запрос + фильтр | `List<HuggingFaceModel>` | Поиск моделей в каталоге HuggingFace |
| `GetModelFilesUseCase` | `UseCase<..., String>` | ID модели | `List<ModelFile>` | Получение списка GGUF-файлов модели |
| `DownloadModelUseCase` | `StreamUseCase<double, DownloadModelParams>` | modelId, filename, startByte | поток прогресса (0.0–1.0) | Скачивание файла модели. Перед началом сохраняет метаданные для возможности возобновления, после завершения — удаляет их |
| `GetLocalModelsUseCase` | `UseCaseNoParams<...>` | — | `List<LocalModel>` | Получение списка скачанных моделей |
| `DeleteModelUseCase` | `UseCase<Unit, String>` | путь к файлу | `Unit` | Удаление модели с устройства |
| `GetPendingDownloadsUseCase` | `UseCaseNoParams<...>` | — | `List<DownloadTaskMetadata>` | Получение списка незавершённых загрузок |
| `GetPartialFileSizeUseCase` | `UseCase<int, String>` | целевой путь | размер в байтах | Определение размера частично скачанного файла (для докачки) |
| `RemoveDownloadMetadataUseCase` | `UseCase<Unit, String>` | имя файла | `Unit` | Удаление метаданных загрузки |

## 6. Слой Data — работа с данными

### 6.1. Сервисы (абстракции и реализации)

Сервисы — низкоуровневые компоненты, каждый из которых отвечает за один источник данных или технологию. Абстракция и реализация разделены для соблюдения **принципа инверсии зависимостей (DIP)**.

#### `LLMService` → `LLMServiceImpl`

**Абстракция** определяет контракт: `loadModel()`, `disposeModel()`, `sendQuery()`, `getModelInfo()`.

**Реализация** работает с пакетом `llamadart`:
- `loadModel()` — создаёт `LlamaEngine` с `LlamaBackend()`, загружает модель по пути, инициализирует `ChatSession`;
- `sendQuery()` — вызывает `session.create([LlamaTextContent(query)])` и генерирует поток строк из `chunk.choices.first.delta.content`;
- `getModelInfo()` — извлекает метаданные через `engine.getMetadata()`, `getContextSize()`, `getBackendName()`, `supportsVision`, `supportsAudio`;
- `disposeModel()` — обнуляет сессию, вызывает `engine.dispose()`.

#### `HuggingFaceApiService` → `HuggingFaceApiServiceImpl`

**Абстракция:** `searchModels()`, `getModelFiles()`, `getDownloadUrl()`.

**Реализация** работает с REST API HuggingFace:
- `searchModels()` — отправляет GET-запрос на `https://huggingface.co/api/models` с параметрами `search`, `sort=downloads`, `limit=20`, фильтром `gguf` (для text-generation) и `pipeline_tag`. Парсит JSON-массив в список `HuggingFaceModel`;
- `getModelFiles()` — запрашивает данные конкретной модели, извлекает `siblings`, фильтрует по расширению `.gguf`, возвращает список `ModelFile`;
- `getDownloadUrl()` — формирует прямую ссылку на скачивание: `https://huggingface.co/{modelId}/resolve/main/{filename}`.

#### `ModelStorageService` → `ModelStorageServiceImpl`

**Абстракция:** `getLocalModels()`, `getDownloadPath()`, `downloadModel()`, `getPartialFileSize()`, `deleteModel()`.

**Реализация** работает с локальной файловой системой:
- `getLocalModels()` — сканирует директорию `models/` в документах приложения, находит `.gguf`-файлы (исключая `.gguf.part` — незавершённые загрузки);
- `downloadModel()` — выполняет потоковую загрузку через `http.Client.send()`. Поддерживает **докачку** через HTTP-заголовок `Range`. Файл сохраняется с суффиксом `.part`, после завершения переименовывается в окончательное имя. Возвращает `Stream<double>` с прогрессом (0.0–1.0);
- `getPartialFileSize()` — проверяет размер файла `.part`, что нужно для определения `startByte` при возобновлении;
- `deleteModel()` — удаляет и основной файл, и `.part`-файл, если он существует.

#### `DownloadMetadataService` → `DownloadMetadataServiceImpl`

**Абстракция:** `getAll()`, `save()`, `remove()`.

**Реализация** хранит метаданные загрузок в JSON-файле `download_metadata.json`:
- `save()` — добавляет или обновляет запись (удаляет старую по `filename`, добавляет новую);
- `remove()` — удаляет запись по `filename`;
- `getAll()` — считывает и десериализует весь файл.

Если загрузка прервалась (закрытие приложения, потеря связи), при следующем запуске `DownloadManagerBloc` считывает этот файл и **возобновляет** все незавершённые загрузки с того байта, на котором они остановились.

### 6.2. Реализации репозиториев

#### `LLMRepositoryImpl`

Оборачивает `LLMService`, делегируя ему все вызовы. Является промежуточным слоем между use cases и сервисом. Сейчас выполняет прямое проксирование, но при необходимости здесь можно добавить кеширование, трансформацию данных или координацию нескольких сервисов.

#### `ModelRepositoryImpl`

Координирует три сервиса: `HuggingFaceApiService`, `ModelStorageService` и `DownloadMetadataService`. Реализует все методы `ModelRepository`:
- Методы поиска и получения файлов делегируются `HuggingFaceApiService`;
- Методы загрузки, хранения и удаления — `ModelStorageService`;
- Методы работы с метаданными загрузок — `DownloadMetadataService`.

Метод `downloadModel()` комбинирует API-сервис (для получения URL) и сервис хранения (для собственно загрузки).

## 7. Слой Presentation — пользовательский интерфейс

### 7.1. BLoC-компоненты

#### `ChatBloc`

**События:**
- `SendMessage(text)` — пользователь отправил сообщение;
- `DisposeModel()` — пользователь покидает экран чата.

**Состояние:** `ChatState` содержит `List<Message> messages` и `bool isGenerating`.

**Логика:** При получении `SendMessage` блок добавляет в список сообщение пользователя и пустое сообщение ассистента, затем подписывается на стрим `LLMSendMessageUseCase`. Каждый полученный фрагмент (`Success`) добавляется в буфер, и обновлённый текст эмитится через `copyWith` последнего сообщения. При ошибке (`Failure`) — в текст записывается сообщение об ошибке. По завершению стрима `isGenerating` устанавливается в `false`.

#### `ModelLoaderBloc`

**События:**
- `LoadModel(modelPath)` — начать загрузку модели в память;
- `UnloadModel()` — выгрузить модель.

**Состояния:** `ModelLoaderInitial` → `ModelLoading` → `ModelLoaded(ModelInfo?)` или `ModelLoadError(error)`.

**Логика:** При `LoadModel` последовательно вызывается `LLMInitUseCase` и `GetModelInfoUseCase`. Если инициализация успешна — эмитится `ModelLoaded` с информацией о модели. В случае ошибки — `ModelLoadError`.

#### `ModelListBloc`

**События:**
- `LoadLocalModels()` — загрузить список локальных моделей;
- `DeleteLocalModel(path)` — удалить модель по пути.

**Состояния:** `ModelListInitial` → `ModelListLoading` → `ModelListLoaded(models)` или `ModelListError(error)`.

**Логика:** При `DeleteLocalModel` после успешного удаления автоматически вызывается `LoadLocalModels` для обновления списка.

#### `ModelSearchBloc`

**События:**
- `SearchModels(query)` — выполнить поиск;
- `ChangePipelineTag(tag)` — сменить фильтр типа модели;
- `ToggleModelExpansion(modelId)` — развернуть/свернуть карточку модели для просмотра файлов.

**Состояние:** `ModelSearchState` — содержит `selectedTag`, `results`, `searching`, `error`, `expandedModelId`, `files`, `loadingFiles`.

**Логика:** При смене тега автоматически повторяется последний поисковый запрос. При разворачивании карточки модели (для `textGeneration`) загружается список GGUF-файлов через `GetModelFilesUseCase`.

**Особенность `copyWith`:** nullable-поля используют паттерн `Function()? wrapper` для различения «не менять» (параметр не передан → `null`) и «установить в null» (передана функция, возвращающая `null`).

#### `DownloadManagerBloc`

**События:**
- `StartDownload(modelId, filename)` — начать загрузку;
- `CancelDownload(filename)` — отменить загрузку;
- `ResumePendingDownloads()` — возобновить все незавершённые загрузки;
- `DownloadProgressUpdated(filename, progress)` — внутреннее событие обновления прогресса;
- `DownloadCompleted(filename)` — внутреннее событие завершения;
- `DownloadFailed(filename, error)` — внутреннее событие ошибки.

**Состояние:** `DownloadManagerState` — содержит `Map<String, DownloadTask> tasks`.

**Логика:** Этот BLoC — **Singleton**, живёт всё время работы приложения. Поддерживает `Map<String, StreamSubscription>` для отслеживания активных загрузок. При `StartDownload` создаёт задачу и подписывается на стрим `DownloadModelUseCase`. Прогресс обновляется через внутренние события. При `CancelDownload` отменяет подписку и удаляет метаданные загрузки. При `ResumePendingDownloads` считывает сохранённые метаданные, определяет размер частично скачанного файла и возобновляет загрузку с нужного байта.

**Метод `close()`:** При закрытии BLoC отменяет все активные подписки на стримы загрузок.

### 7.2. Страницы (Pages)

#### `App` — корневой виджет

Оборачивает приложение в `BlocProvider.value` для глобального `DownloadManagerBloc`. Настраивает `MaterialApp` с Material 3, задаёт начальный маршрут `/` (ModelListPage) и карту маршрутов.

#### `ModelListPage` — список локальных моделей (главный экран)

- Отображает список скачанных моделей (`LocalModelTile`);
- Вверху показывает активные загрузки (`DownloadProgressTile`);
- При пустом списке — заглушка `EmptyModelList` с кнопкой перехода к поиску;
- Поддерживает pull-to-refresh через `RefreshIndicator`;
- FAB-кнопка открывает экран поиска;
- `BlocListener` на `DownloadManagerBloc` автоматически обновляет список при завершении загрузки;
- Нажатие на модель — переход на `LoadingPage` с передачей пути к файлу.

#### `ModelSearchPage` — поиск моделей на HuggingFace

- Текстовое поле с **debounce** (500 мс) — задержка перед отправкой запроса;
- Фильтр по типу задачи через `PipelineTagChips` (горизонтальный список чипов);
- Результаты отображаются как `ModelSearchCard` с возможностью развернуть и увидеть список файлов;
- Для моделей `textGeneration` — список GGUF-файлов с кнопками скачивания;
- Для других типов — надпись «Support coming soon»;
- SnackBar при завершении загрузки.

#### `LoadingPage` — экран загрузки модели в память

- Получает путь к модели через `ModalRoute.settings.arguments`;
- Создаёт `ModelLoaderBloc` и сразу отправляет событие `LoadModel(modelPath)`;
- Показывает индикатор загрузки или ошибку с кнопкой повтора;
- При успешной загрузке через `BlocListener` выполняет навигацию на `ChatPage`, передавая `ModelInfo` в аргументах.

#### `ChatPage` — экран чата с моделью

- Создаёт `ChatBloc` через `BlocProvider`;
- Получает `ModelInfo` из аргументов навигации;
- `AppBar` с кнопкой «назад» (выгружает модель и возвращает на главный экран) и кнопкой «info» (показывает `ModelInfoSheet`);
- Тело — `ChatBody` с `ListView` сообщений;
- Внизу — панель ввода с `TextField`, кнопкой отправки и индикатором генерации;
- `BlocSelector` используется для оптимизации: кнопка отправки перестраивается только при изменении `isGenerating`;
- Автоматическая прокрутка к последнему сообщению через `_scrollToBottom()`.

### 7.5. Утилиты и расширения

#### `Constants`
Класс со всеми строковыми константами приложения: названия кнопок, сообщения об ошибках, подсказки. Централизация строк упрощает поддержку и будущую локализацию.

#### `AppDimens`
Класс с числовыми константами: отступы (`spacingSm`–`spacingXxl`), радиусы (`radiusSm`–`radiusLg`), размеры иконок, параметры теней. Вынесение размеров из виджетов обеспечивает единообразие UI.

#### `FileSizeExtension`
```dart
extension FileSizeExtension on int {
  String get sizeFormatted { ... }
}
```
Расширение для типа `int`, форматирующее размер в байтах в человекочитаемый формат (GB/MB). Используется в `LocalModelTile` и `ModelFileTile`.
